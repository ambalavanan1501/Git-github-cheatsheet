01 Key Concepts What every term means
🗂 Repository (Repo)
A folder tracked by Git. Contains all your files + the full history of changes.
📸 Commit
A saved snapshot of your files at a moment in time. Like a game checkpoint.
🌿 Branch
A parallel version of your code. Work on features without touching main code.
🔀 Merge
Combine changes from one branch into another.
🌐 Remote
A copy of your repo hosted online (e.g. GitHub). Called origin by default.
📤 Push
Upload your local commits to the remote repo on GitHub.
📥 Pull
Download changes from the remote and merge them into your local branch.
🧩 Clone
Make a local copy of a remote repo. Only done once per project.
🎭 Staging Area
A waiting room for changes. You add files here before committing.
🔁 Pull Request (PR)
A GitHub proposal to merge one branch into another. Enables code review.
⚔️ Merge Conflict
When two branches changed the same line. Git asks you to choose which to keep.
🏷 Tag
A named marker on a commit, usually for releases like v1.0.0.

03 Basic Commands Your daily workflow
🚀 Starting a Repo
  • git initCreate a new local repo in current foldercd my-project && git init
  • git clone <url>Copy a remote repo to your machinegit clone https://github.com/user/repo.git
  • git clone <url> my-folderClone into a specific folder name
➕ Staging Files
  • git add <file>Stage one specific filegit add index.html
  • git add .Stage ALL changed files in current folder
  • git add *.jsStage all JavaScript files
  • git restore --staged <file>Unstage a file (keep changes)
📸 Committing
  • git commit -m "message"Save staged changes with a messagegit commit -m "add login page"
  • git commit -am "message"Stage tracked files AND commit in one step
  • git commit --amendEdit the last commit's message or files
Good commit messagesUse present tense: "add feature" not "added feature". Format: feat:, fix:, docs:
🔍 Viewing History

  • git log --oneline --graphHistory as a visual branch graph

🙈 .gitignore
  • .gitignoreFile that tells Git which paths to never track

04 Branching & Merging Work in parallel safely
🌿 Branch Diagram
Branches let you develop features independently. Merge them back to main when done.
🌿 Branch Commands
  • git branchList all local branches (* = current)
  • git branch <name>Create a new branchgit branch feature/login
  • git switch <name>Switch to a branch
  • git switch -c <name>Create AND switch in one step ⭐git switch -c feature/login
  • git branch -d <name>Delete branch (safe — won't delete unmerged)
  • git branch -D <name>Force-delete a branch
  • git branch -m <old> <new>Rename a branch
🔀 Merging
  • git merge <branch>Merge a branch INTO your current branchgit switch main then git merge feature/login
  • git merge --no-ff <branch>Force a merge commit (keeps history clean)
  • git merge --abortCancel a merge with conflicts
Merge ConflictsWhen Git can't auto-merge, it marks the file. Open it, choose which code to keep, then git add and git commit.
♻️ Rebasing
  • git rebase mainMove your branch's commits on top of main (linear history)
  • git rebase --abortCancel a rebase
  • git rebase --continueAfter resolving conflicts, continue rebase
Merge vs RebaseMerge keeps full history. Rebase makes history linear. Beginners: use merge.

🔗 Managing Remotes
  • git remote -vList all remotes and their URLs
  • git remote add origin <url>Link your local repo to GitHubgit remote add origin https://github.com/u/repo.git
  • git remote remove originRemove a remote connection
  • git remote set-url origin <url>Change the remote URL
📤 Pushing
  • git pushPush current branch to remote
  • git push origin mainPush main branch to origin
  • git push -u origin <branch>Push + set upstream (first push of a branch) ⭐git push -u origin feature/login
  • git push --forceForce push ⚠️ overwrites remote history
  • git push origin --delete <branch>Delete a remote branch
📥 Pulling & Fetching
  • git fetchDownload changes but don't merge
  • git fetch originFetch from origin remote specifically
  • git pullFetch + merge into current branch ⭐
  • git pull origin mainPull from main branch of origin
  • git pull --rebasePull with rebase instead of merge
fetch vs pullfetch = safe preview. pull = fetch + auto-merge.
06 Undoing Mistakes Every developer needs this
↩️ Restore & Reset
  • git restore <file>Discard changes in working dir (undo edits)git restore index.html
  • git restore --staged <file>Unstage file (keep changes)
  • git reset --soft HEAD~1Undo last commit, keep changes staged
  • git reset --mixed HEAD~1Undo last commit, keep changes unstaged
  • git reset --hard HEAD~1⚠️ Undo last commit AND discard all changes
  • git clean -fd⚠️ Delete all untracked files and folders
🔄 Revert & Stash
  • git revert <hash>Create a new commit that undoes a commit (safe for shared repos)git revert a3f9d12
  • git stashTemporarily save uncommitted work
  • git stash popRestore stashed work
  • git stash listSee all stashes
  • git stash dropDelete the latest stash
--hard is permanentgit reset --hard destroys your changes forever. Use with extreme care.
🗺 Which "undo" command should I use?
SituationCommandSafe?
I edited a file and want to undo all editsgit restore <file>✅ Yes
I staged a file but don't want to commit itgit restore --staged <file>✅ Yes
I committed but want to redo the commit messagegit reset --soft HEAD~1✅ Yes
I committed something wrong (on my own branch)git reset --hard HEAD~1⚠️ Loses data
I need to undo a commit on a shared branchgit revert <hash>✅ Yes
I need to save work before switching branchesgit stash✅ Yes

10 Beginner Tips & Best Practices Learn from others' mistakes
Commit often, push oftenSmall commits are easier to understand, revert, and review.
Use meaningful branch namesUse feature/login, fix/navbar-crash, chore/update-deps — not my-branch.
Pull before you pushAlways git pull before pushing to avoid conflicts on shared branches.
Never commit secretsPut API keys, .env files, and passwords in .gitignore. Once committed, they're in history forever.
Write good commit messagesFormat: type: short description e.g. feat: add dark mode, fix: fix login crash
Don't force push to shared branchesgit push --force on main can erase teammates' work permanently.
Avoid committing node_modulesAlways add node_modules/ to .gitignore. It can be 100k+ files.
Use SSH instead of HTTPSSet up SSH keys so you don't need to enter your password on every push.
Keep main cleanOnly merge working, reviewed code into main. Always work on branches.
GitHub browser shortcutPress . on any GitHub repo to open VS Code in your browser!
